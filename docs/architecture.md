### Key User Flows

**Flow 1 — Subscriber discovers and downloads a dataset**
1. User logs in at `/login` → Cognito JWT issued → stored in `localStorage` via `auth.store.ts`
2. Dashboard (`/dashboard`) loads — TanStack Query fetches `/api/v1/me` and KPI tiles in parallel
3. User browses `/catalog` → server-side paginated search with filters → clicks dataset card
4. Dataset detail (`/catalog/:id`) → user clicks "Download" → POST `/api/v1/datasets/:id/download` returns `job_id`
5. Frontend polls `GET /api/v1/jobs/:id` with `refetchInterval` until `status === 'completed'` → download link rendered

**Flow 2 — Org Admin manages team entitlements**
1. Admin navigates to `/org` → fetches team members and org entitlements via TanStack Query
2. Invites member by email → POST `/api/v1/org/members` → MSW handler updates optimistically
3. Assigns dataset access → PATCH entitlement → invalidates query cache on success

**Flow 3 — API Developer generates and rotates a key**
1. Navigate to `/api-access` → list of API keys fetched → usage graph rendered (7-day window)
2. Create new key (named) → POST `/api/v1/api-keys` → key displayed once, then only hash stored
3. Revoke old key → DELETE → list refetches

## 2. AWS Infrastructure

### Services

| Service | Role | Justification |
|---------|------|---------------|
| **CloudFront** | CDN + HTTPS termination for the SPA | Low-latency global delivery; cache immutable assets (JS/CSS) with `Cache-Control: max-age=31536000`, HTML with `no-cache` |
| **GitHub Pages** | Static origin for SPA build artefacts | Already in scaffold (`deploy.target: github-pages`); CloudFront sits in front for custom domain + TLS |
| **Amazon Cognito** | User authentication + JWT issuance | Direct user pool integration (no Amplify); supports TOTP MFA, hosted UI optional, JWT validated at API GW |
| **API Gateway (HTTP API)** | REST entry point for all `/api/v1/*` calls | JWT authoriser validates Cognito tokens; no long-lived keys in frontend code |
| **S3** | Dataset file storage | Server-side encryption (SSE-KMS); presigned URLs expire after 24 h — never expose raw bucket URLs to client |
| **RDS PostgreSQL** | Relational store for all domain entities | Multi-AZ for HA; encrypted at rest |
| **CloudWatch + X-Ray** | Observability | Structured logs from backend Lambda/ECS; p99 latency alarms; X-Ray traces for download job pipeline |

### Region & Multi-AZ

- Primary region: `ap-southeast-1` (Singapore) — closest to TWA's data publisher base
- Multi-AZ: RDS with standby replica; ALB/API GW span all AZs in region
- CloudFront: global edge — no regional constraint for static assets

### VPC Layout

```
ap-southeast-1
├── Public subnets (a, b, c)
│   └── ALB / API Gateway endpoints
└── Private subnets (a, b, c)
    ├── ECS / Lambda (backend services)
    └── RDS PostgreSQL (Multi-AZ)

NAT Gateway in public subnet → private subnet internet egress (npm installs, Cognito calls)
S3, Cognito accessed via VPC endpoints (no NAT traversal)
```

### OIDC Deploy Role

GitHub Actions assumes a deploy role via OIDC — no long-lived AWS keys committed or stored in GitHub secrets. The role is scoped to `PutObject` on the Pages S3 bucket and `CreateInvalidation` on the CloudFront distribution.

## 4. API Contract Overview

**Base path:** `https://api.twa.io/api/v1`
**Versioning:** URI prefix (`/v1`); breaking changes → `/v2` with 6-month deprecation window

### Auth

All endpoints (except `/auth/*`) require:
```
Authorization: Bearer <Cognito access token>
```
API Gateway JWT authoriser validates:
- Issuer: `https://cognito-idp.ap-southeast-1.amazonaws.com/<user_pool_id>`
- Audience: Cognito App Client ID
- Expiry: 15-minute TTL; frontend auto-refreshes via `shared/auth.ts → refreshSession()`

### Endpoint Groups

| Group | Paths | Notes |
|-------|-------|-------|
| **Auth** | `/auth/login`, `/auth/refresh`, `/auth/mfa/verify` | Cognito proxy; no auth header needed |
| **Datasets** | `/datasets`, `/datasets/:id`, `/datasets/:id/download`, `/datasets/:id/preview` | Entitlement check on download |
| **Jobs** | `/jobs/:id` | Poll until `status === 'completed'`; use `refetchInterval` |
| **API Keys** | `/api-keys`, `/api-keys/:id` | Key shown once; bcrypt-hashed at rest |
| **Usage** | `/usage` | Caller's own stats; rate limited |
| **Org** | `/org/members`, `/org/entitlements` | ORG_ADMIN role required |
| **Admin** | `/admin/users`, `/admin/datasets`, `/admin/audit` | ADMIN role required |
| **Billing** | `/subscriptions`, `/invoices` | Read org plan; upgrade/downgrade |

### Error Shape

```json
{
  "code": "ENTITLEMENT_REQUIRED",
  "message": "Your organisation does not have access to this dataset.",
  "request_id": "req_01hw..."
}
```

HTTP status codes: 400 validation, 401 unauthenticated, 403 forbidden, 404 not found, 429 rate limited, 500 internal.

## 5. Security Model

### Cognito JWT Validation

- Access token TTL: 15 minutes
- Refresh token TTL: 30 days
- Frontend stores tokens in `localStorage` via `store/auth.store.ts`; never in cookies or `sessionStorage`
- 401 from any API call → clear tokens → redirect `/login`
- Token refresh handled by `shared/auth.ts → refreshSession()` before expiry

### RBAC

| Role | Dashboard | Catalog | Downloads | API Access | Org Admin | Admin Panel |
|------|-----------|---------|-----------|------------|-----------|-------------|
| SUBSCRIBER | ✓ | ✓ | ✓ | ✓ | — | — |
| ORG_ADMIN | ✓ | ✓ | ✓ | ✓ | ✓ | — |
| PUBLISHER | ✓ | ✓ | — | — | — | — |
| ADMIN | ✓ | ✓ | ✓ | ✓ | ✓ | ✓ |

Role is read from the Cognito JWT `custom:role` claim; enforced server-side on every request. Frontend hides nav items by role but never trusts client-side gating as a security boundary.

### Encryption

- **In transit:** TLS 1.2+ enforced at CloudFront and API Gateway; HSTS header set
- **At rest:** RDS encrypted (AES-256 via RDS key); S3 datasets SSE-KMS; API key hashes (bcrypt, cost factor 12)
- **Presigned URLs:** expire 24 h; scoped to single object; no wildcard permissions

### OWASP ASVS

Target: **Level 2** (standard-assurance applications handling sensitive commercial data)

Key controls:
- Input validation at API boundary (Yup on frontend, server-side validation on backend — never trust client)
- No sensitive data in URL query params (tokens, keys)
- `Content-Security-Policy` header on all responses
- Rate limiting: per-API-key daily quota at API Gateway; 429 on breach
- Audit log: every login, download, key action, admin operation logged with actor + IP + timestamp

## 6. Non-Functional Requirements

| Concern | Target | Approach |
|---------|--------|----------|
| **Availability** | 99.9% uptime | CloudFront serves static SPA from cache even if origin is down; API Gateway + Multi-AZ RDS for backend |
| **Latency** | Dashboard p95 < 2 s | Parallel TanStack Query fetches on mount; skeleton loaders; CDN for assets; server-side pagination for catalog |
| **Download throughput** | Support files up to 10 GB | Presigned S3 URLs — browser downloads directly from S3, not through API; no proxy bottleneck |
| **Scalability** | 10× burst without config change | CloudFront absorbs static traffic; API Gateway + Lambda/ECS autoscales; RDS connection pooling (PgBouncer) |
| **Observability** | Full trace per request | CloudWatch structured logs; X-Ray trace from API GW → backend → RDS; RUM (CloudWatch or Datadog) for frontend p95 |
| **Search performance** | Catalog search < 500 ms p99 | PostgreSQL full-text search with `GIN` index on `dataset.name + description + variable_tags`; Redis cache for popular queries (future) |

## 7. Frontend Architecture

### Module Structure

```
src/
  components/       Shared primitives (Button, Badge, DataTable, SkeletonCard …)
  modules/
    auth/           Login, MFA challenge, password reset, session refresh
    dashboard/      KPI tiles, recent datasets, notification bell
    catalog/        Search bar, filter sidebar, dataset cards, list/grid toggle
    dataset/        Detail page, map preview, download trigger, version history
    downloads/      Jobs table, status polling, download link
    api-access/     Key management, usage graph, API explorer embed
    org/            Member list, entitlement view, invite flow
    admin/          User management, dataset moderation, audit log, health tiles
    subscription/   Plan card, upgrade/downgrade, invoice history
    profile/        Edit profile, MFA toggle, saved searches
  pages/            Thin route shells — one per route
  shared/           ApiClient (Axios), auth helpers, env, logger
  store/            auth.store.ts (Zustand), ui.store.ts
  styles/           globals.css (Tailwind entry)
  types/            Shared TypeScript interfaces
  mocks/            MSW handlers + browser/server setup
```

### State Strategy

| Concern | Tool | Why |
|---------|------|-----|
| Server state (datasets, jobs, org) | TanStack Query | Caching, background refetch, polling (`refetchInterval` for jobs), optimistic updates |
| Auth state (tokens, user, role) | Zustand `auth.store` | Persisted to `localStorage`; read by `ApiClient` to inject Bearer header |
| UI state (modals, filters, view mode) | Zustand `ui.store` or local `useState` | Ephemeral; no persistence needed |

### MSW Mock Strategy

- `VITE_USE_MOCK=true` activates browser service worker (`src/mocks/browser.ts`) in dev
- Tests always use Node MSW server (`tests/setup.ts`) — no env var needed
- Handlers mirror the API contract: one handler file per domain (`auth.ts`, `datasets.ts`, `jobs.ts`, `api-keys.ts`, `org.ts`, `admin.ts`)
- Health check handler already in place (`GET /api/v1/health`)
